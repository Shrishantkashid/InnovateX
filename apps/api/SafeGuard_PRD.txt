================================================================================

SafeGuard is an AI-powered safety platform with three interconnected modules
targeting women, children who carry phones, and children's digital activity.

The core thesis: every existing safety app requires the victim to act. SafeGuard
is the first system that protects users when they cannot act — using passive AI
that listens, watches, and responds automatically.

2. PROBLEM STATEMENT
================================================================================

Women and children face physical and digital safety threats daily. Despite
dozens of existing safety apps, three critical gaps remain unsolved:

GAP 1 — REQUIRES USER ACTION
All existing SOS apps (bSafe, Shake2Safety, Safetipin, Noonlight) require the
user to physically tap, hold, or shake their phone. In real danger, victims
are often physically restrained, in shock, or cannot reach their device.

GAP 2 — REACTIVE NOT PREDICTIVE
Current apps respond after danger is reported. None analyse route risk before
the journey starts, or detect threat signals before an incident occurs.

GAP 3 — NO CHILD-SPECIFIC DIGITAL THREAT DETECTION
Parental control apps (Qustodio, Screen Time, Family Link) either block
everything bluntly or screenshot messages invasively. None use AI to detect
grooming patterns in conversation behaviour without reading message content.

SafeGuard addresses all three gaps across three user modules.

6. MODULE 1 — WOMEN PHYSICAL SAFETY
================================================================================

6.1 OVERVIEW
------------
Passive, automatic threat detection that works even when the woman cannot
physically interact with her phone.

6.2 USER JOURNEY
----------------
Step 1 — Plan journey
  User enters destination. AI scores route risk (1–5) using time-of-day,
  historical crime data, and Google Places. Suggests safest path.

Step 2 — Journey mode ON
  App runs silently in background. Microphone monitors for threat audio.
  Accelerometer monitors for struggle/impact. No interaction required.

Step 3 — Threat detected
  AI detects distress audio OR motion anomaly OR missed ETA.
  Sends silent 10-second confirmation ping to user.

Step 4 — No response
  If user doesn't dismiss: auto-SOS fires.
  Live location + 30s audio clip sent to contacts + authority dashboard.

Step 5 — Safe arrival
  User taps "I'm home safely." Contacts notified. Trip data fed back
  to improve area risk model.

6.3 FEATURE SET
---------------
FEATURE: Journey Guardian Mode
  Description : User sets destination + ETA. If not dismissed on arrival,
                auto-SOS fires after 60s grace period.
  Tech        : Background timer + expo-location geofencing
  Priority    : Must build (hours 0–10)

FEATURE: Passive Audio Sentinel
  Description : Microphone samples 3s audio every 10s. YAMNet TFLite
                classifies for screaming, confrontation, sudden silence.
                Trigger threshold: confidence >= 0.70
  Tech        : expo-av + YAMNet TFLite (on-device inference)
  Priority    : Must build (hours 0–10)

FEATURE: SOS with Audio Evidence
  Description : On trigger (auto or manual), app records last 30s of audio
                + current GPS. Sends both to trusted contacts and stores in
                Supabase Storage for authority evidence.
  Tech        : expo-av circular buffer + Supabase Storage
  Priority    : Must build (hours 0–10)

FEATURE: Route Risk Intelligence
  Description : Before journey, GPT-4o-mini analyses destination + time of
                day + historical crime CSV → returns risk score 1–5 with
                plain-English reasoning + safer alternative route.
  Tech        : GPT-4o-mini + Google Places API + NCRB crime data CSV
  Priority    : Differentiator (hours 10–18)

FEATURE: Silent Codeword Trigger
  Description : User pre-registers a phrase (e.g. "call mom"). On-device
                STT listens continuously. If phrase detected → SOS fires
                silently. No UI change visible to attacker.
  Tech        : expo-speech / Whisper on-device STT
  Priority    : Differentiator (hours 10–18)

FEATURE: Motion Anomaly Detection
  Description : Accelerometer sudden spike (impact) OR gyroscope rapid
                rotation OR phone stationary for 5+ min post-SOS → escalate
                alert tier automatically.
  Tech        : expo-sensors (accelerometer + gyroscope fusion)
  Priority    : Differentiator (hours 10–18)

FEATURE: Fake Call Screen
  Description : 2-tap combo opens a fake incoming call UI over the home
                screen. Gives user a plausible cover action.
  Tech        : React Native modal overlay
  Priority    : Wow factor (hours 18–22)

FEATURE: Community Guardian Network
  Description : Nearby verified users within 500m receive a push notification
                when SOS fires: "Someone nearby needs help."
  Tech        : PostGIS geo-query + Expo Push
  Priority    : Wow factor (hours 18–22)

6.4 THREAT FUSION THRESHOLDS (MODULE 1)
-----------------------------------------
  Audio confidence    >= 0.70  → flag
  Motion spike        >= 2.5g  → flag
  ETA overdue         >= 5 min → flag
  Time of day weight  : 10pm–5am = +15 pts to score

  Score 0–40   : Monitor (no action)
  Score 40–70  : Ping user with 10s dismiss window
  Score 70–100 : Auto-SOS fires

7. MODULE 2 — CHILD PHYSICAL SAFETY
================================================================================

7.1 OVERVIEW
------------
Same threat model as Module 1, but child-specific: lower trigger thresholds,
faster escalation, parent-first notification order, trusted adult radius system,
and squeeze gesture for discreet SOS without codeword memory.

7.2 KEY DIFFERENCES FROM MODULE 1
-----------------------------------
  Threshold          : 0.55 confidence (vs 0.70 for women) — children
                       show less extreme acoustic distress signals
  Confirm window     : 5 seconds (vs 10 seconds)
  Escalation order   : Parent auto-call → trusted adult → contacts → authority
                       (women: contacts → authority)
  SOS trigger        : Squeeze gesture (volume up + down × 3) — works with
                       phone in pocket, no codeword needed
  No codeword        : Child doesn't need to remember or speak calmly
  Screen change      : None on SOS — attacker sees nothing happen

7.3 TRUSTED ADULT RADIUS SYSTEM
---------------------------------
  Setup (parent does this during onboarding):
    - Draw safe zones on map: school, home, tutor's house, friend's home
    - Add trusted adult list: teacher name+phone, aunt, neighbour
    - Set school route schedule (time + expected path)

  Runtime behaviour:
    - App continuously checks child GPS against defined safe zones
    - If child exits ALL zones AND no active "on journey" tag:
        → Soft alert sent to parent with live location (not full SOS)
        → Parent has 90 seconds to respond
        → If no response → escalates to full SOS chain

7.4 FEATURE SET
---------------
FEATURE: Squeeze SOS Gesture
  Description : Volume up + volume down pressed 3 times in quick succession.
                Works with phone in pocket, screen off. Silent — no sound,
                no screen change, no vibration visible to attacker.
  Tech        : React Native volume button listener (expo-av)
  Priority    : Must build

FEATURE: Trusted Radius Engine
  Description : PostGIS polygon zones + real-time GPS check. Zone exit
                triggers soft alert chain.
  Tech        : PostGIS ST_Within() query, Supabase Realtime, expo-location
  Priority    : Must build

FEATURE: Parent Live View
  Description : Parent app shows child's real-time location on Leaflet map.
                Toggle to listen to ambient audio (parent must explicitly enable).
  Tech        : Supabase Realtime + Leaflet.js
  Priority    : Must build

FEATURE: Audio Sentinel (Child-Tuned)
  Description : Same YAMNet pipeline as Module 1, threshold lowered to 0.55.
                Confirm window 5s (vs 10s).
  Tech        : YAMNet TFLite on-device
  Priority    : Must build

FEATURE: ETA Guardian
  Description : Parent sets expected arrival time for school, tuition, etc.
                If child doesn't reach geofenced destination by ETA → soft alert.
  Tech        : expo-location geofence + background timer
  Priority    : Must build

FEATURE: Parent-First Escalation
  Description : On SOS fire:
                T+0s   : Parent auto-call (Twilio Voice)
                T+30s  : Trusted adult push notification (if parent unreachable)
                T+60s  : All contacts SMS + live link
                T+90s  : Authority dashboard incident created
  Tech        : Twilio Voice + Expo Push + Supabase
  Priority    : Must build

7.5 THREAT FUSION THRESHOLDS (MODULE 2)
-----------------------------------------
  Audio confidence    >= 0.55  → flag (lower than M1)
  Motion spike        >= 2.0g  → flag (lower than M1)
  Zone exit           = instant soft alert
  ETA overdue         >= 3 min → flag (shorter grace than M1)
  Time of day weight  : applies same as M1

  Score 0–30   : Monitor
  Score 30–55  : Parent ping (shorter window — children need faster response)
  Score 55–100 : Auto-SOS fires

8. MODULE 3 — DIGITAL CHILD SAFETY (PARENTAL CONTROLS)
================================================================================

8.1 OVERVIEW
------------
AI-powered digital safety that detects grooming patterns, behavioural shifts,
and content risks — without reading message content. Privacy-preserving by
design so the child doesn't resent the system and uninstall it.

8.2 CORE PHILOSOPHY
-------------------
"We don't read your child's messages. We watch how conversations behave."

The on-device monitor agent extracts only metadata and pattern scores —
no raw message text is ever uploaded to the server. The MiniLM classifier
runs entirely on the child's device. Only a scored, encrypted payload is
sent to the backend.

8.3 THREE DETECTION PILLARS
-----------------------------

PILLAR 1: GROOMING PATTERN DETECTOR
  What it detects (behavioural signals, not message content):
    - New contact (not in phone contacts) + adult name pattern
    - Rapid message frequency escalation within 48 hours
    - Secrecy cluster: "don't tell", "our secret", "just between us"
    - Platform migration: "add me on Snapchat / Signal instead"
    - Gift language: "I'll send you", "I can get you", "do you want"
    - Late-night messaging window with unknown contact
    - Age gap inference from name + conversation patterns

  How it works:
    MiniLM (lightweight NLP) runs on-device. It classifies conversation
    metadata + flagged phrase clusters into a per-contact risk score 0–100.
    No message text is transmitted — only {contact_id, flag_type, confidence}.

  Model: sentence-transformers/paraphrase-MiniLM-L3-v2 (quantized, ~17MB)
  Inference: on-device Python (Termux) or React Native bridged module
  Output: POST /ai/grooming-score {contact_id, score, flags[]}

PILLAR 2: BEHAVIOUR SHIFT TRACKER
  What it detects:
    - Baseline: 4-week rolling average of app usage patterns
    - Flags: sudden increase in late-night phone use
    - Flags: switching apps when parent is nearby (detected via motion)
    - Flags: surge in messages from new unknown contact
    - Flags: sudden drop in family communication apps

  How it works:
    Usage log collected every 15 minutes. Python worker compares current
    week vs 4-week baseline. Delta > 2 standard deviations = flag.

  Tech: Python statsmodels zscore + Supabase behaviour_baselines table
  Output: POST /ai/behaviour-delta {child_id, delta_score, anomalies[]}

PILLAR 3: CONTENT RISK SCORER
  What it detects:
    - NSFW image classification results (on-device, score only uploaded)
    - DNS block event log (which categories triggered safe search)
    - Search term clusters (hashed — specific terms never stored)

  How it works:
    On-device NSFW classifier (MobileNet-based, ~8MB) scores images
    received in gallery. Score only (0–1 float) is uploaded — not the image.
    DNS filter events come from NextDNS API integration.

  Tech: TFLite NSFW model + NextDNS API
  Output: POST /ai/content-flags {child_id, nsfw_score, dns_categories[]}

8.4 RISK SYNTHESIS ENGINE
--------------------------
  Inputs: grooming_score + behaviour_delta + content_flags
  Model : GPT-4o-mini with structured JSON output
  Output:
    {
      composite_risk: 0–100,
      risk_category: "low"|"moderate"|"high"|"critical",
      top_flags: string[],
      weekly_narrative: string,   // plain English summary for parent
      conversation_starter: string // how to talk to child about this
    }

  Runs: daily batch (cron) + immediately on critical flag

8.5 PARENT INTELLIGENCE DASHBOARD
-----------------------------------
  Risk score card       : Per-contact composite score, category breakdown,
                          trend sparkline over 30 days
  Weekly AI digest      : Plain-language narrative — no raw logs shown
                          "Your child had 3 conversations with an unknown
                          contact this week showing secrecy patterns."
  Conversation coach    : GPT-4o-mini generated guide: "Here's a natural
                          way to bring this up without breaking trust."
  Instant alerts        : Push notification on high/critical risk flag
  App usage charts      : Per-app daily time (Recharts bar chart)

8.6 PARENTAL CONTROL ENFORCEMENT
----------------------------------
  DNS-level safe search : NextDNS API — blocks adult content at network level
  Per-app time caps     : Daily limits enforced via device management profile
  Contact allowlist     : Unknown contacts auto-flagged if not on parent list
  Late-night lock       : Schedule-based phone lock (10pm–6am configurable)
  Screen time report    : Weekly PDF digest emailed to parent

  Framework    : FastAPI (Python 3.11)

  Realtime     : Live location streaming, SOS event push
  Storage      : Audio evidence clips, PDF digests
  PostGIS      : Geo-queries for trusted radius, guardian network
  RLS          : Row-level security — child can only read own rows;
                 parent can read child rows they are linked to

9.3 REDIS (OPTIONAL LAYER)
---------------------------
  Use case     : Rate-limit SOS events per user (prevent spam)
                 Cache route risk scores (60-min TTL)
                 Session tokens for public tracking links

10. DATABASE SCHEMA (ALL TABLES)
================================================================================

--- SHARED TABLES ---

TABLE: users
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  email           text UNIQUE NOT NULL
  phone           text
  full_name       text
  role            text CHECK (role IN ('woman','child','parent','admin'))
  module          text CHECK (module IN ('m1','m2','m3'))
  created_at      timestamptz DEFAULT now()

TABLE: trusted_contacts
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  user_id         uuid REFERENCES users(id)
  contact_name    text NOT NULL
  contact_phone   text NOT NULL
  contact_email   text
  is_trusted_adult boolean DEFAULT false
  created_at      timestamptz DEFAULT now()

TABLE: notification_log
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  user_id         uuid REFERENCES users(id)
  sos_event_id    uuid
  channel         text  -- 'sms'|'whatsapp'|'voice'|'push'|'email'
  recipient_phone text
  status          text  -- 'sent'|'delivered'|'failed'
  sent_at         timestamptz DEFAULT now()

--- MODULE 1 + 2 TABLES ---

TABLE: sos_events
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  user_id         uuid REFERENCES users(id)
  module          text CHECK (module IN ('m1','m2'))
  trigger_type    text  -- 'audio'|'motion'|'eta'|'squeeze'|'manual'|'zone_exit'
  threat_score    int   -- 0–100
  lat             float8
  lng             float8
  audio_clip_url  text  -- Supabase Storage URL
  track_token     text  -- public tracking page token
  resolved        boolean DEFAULT false
  created_at      timestamptz DEFAULT now()

TABLE: location_pings
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  user_id         uuid REFERENCES users(id)
  sos_event_id    uuid REFERENCES sos_events(id)
  lat             float8 NOT NULL
  lng             float8 NOT NULL
  accuracy        float8
  recorded_at     timestamptz DEFAULT now()

TABLE: audio_clips
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  user_id         uuid REFERENCES users(id)
  sos_event_id    uuid REFERENCES sos_events(id)
  storage_url     text NOT NULL
  duration_sec    int
  yamnet_class    text   -- top YAMNet classification label
  yamnet_score    float8
  created_at      timestamptz DEFAULT now()

TABLE: trusted_zones  (Module 2 only)
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  child_id        uuid REFERENCES users(id)
  label           text  -- 'school'|'home'|'tutor'|'custom'
  polygon         geometry(Polygon, 4326)  -- PostGIS polygon
  active_days     int[] -- [1,2,3,4,5] = Mon–Fri
  active_start    time
  active_end      time
  created_at      timestamptz DEFAULT now()

TABLE: guardian_pings
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  sos_event_id    uuid REFERENCES sos_events(id)
  guardian_user_id uuid REFERENCES users(id)
  distance_m      float8
  responded       boolean DEFAULT false
  created_at      timestamptz DEFAULT now()

--- MODULE 3 TABLES ---

TABLE: contact_risk_scores
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  child_id        uuid REFERENCES users(id)
  contact_hash    text  -- hashed contact ID, not raw phone
  grooming_score  float8
  behaviour_delta float8
  content_score   float8
  composite_score float8
  risk_category   text  -- 'low'|'moderate'|'high'|'critical'
  top_flags       text[]
  scored_at       timestamptz DEFAULT now()

TABLE: behaviour_baselines
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  child_id        uuid REFERENCES users(id)
  app_name        text
  avg_daily_min   float8   -- 4-week baseline in minutes
  late_night_freq float8   -- avg late-night sessions per week
  updated_at      timestamptz DEFAULT now()

TABLE: app_usage_log
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  child_id        uuid REFERENCES users(id)
  app_name        text
  session_start   timestamptz
  session_end     timestamptz
  duration_min    float8
  is_late_night   boolean

TABLE: weekly_digests
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  child_id        uuid REFERENCES users(id)
  parent_id       uuid REFERENCES users(id)
  narrative       text   -- GPT-4o-mini plain English summary
  conversation_starter text
  composite_risk  float8
  pdf_url         text   -- Supabase Storage
  week_start      date
  created_at      timestamptz DEFAULT now()

TABLE: content_flags
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
  child_id        uuid REFERENCES users(id)
  flag_type       text  -- 'nsfw_image'|'dns_block'|'search_cluster'
  score           float8
  category        text
  flagged_at      timestamptz DEFAULT now()

11. API SPECIFICATION (ALL ENDPOINTS)
================================================================================

BASE URL: https://api.safeguard.app (prod) | http://localhost:8000 (dev)

All endpoints require:
  Header: Authorization: Bearer <jwt_token>
  Header: Content-Type: application/json
  Header: X-Module: m1|m2|m3

--- SOS & LOCATION ---

POST /api/sos
  Description  : Fire an SOS event (auto or manual trigger)
  Auth         : Required (woman or child role)
  Request body :
    {
      "user_id": "uuid",
      "lat": 12.9716,
      "lng": 77.5946,
      "trigger_type": "audio|motion|eta|squeeze|manual|zone_exit",
      "threat_score": 78,
      "audio_clip_b64": "base64string|null"
    }
  Response 200 :
    {
      "sos_event_id": "uuid",
      "track_token": "abc123",
      "track_url": "https://safeguard.app/track/abc123",
      "notifications_queued": 3
    }
  Side effects : Creates sos_events row, queues notification worker,
                 uploads audio to Supabase Storage, pings guardian network

POST /api/location
  Description  : Stream live location during active SOS
  Auth         : Required
  Request body : { "user_id": "uuid", "sos_event_id": "uuid",
                   "lat": float, "lng": float, "accuracy": float }
  Response 200 : { "received": true }
  Side effects : Inserts location_pings row, broadcasts via Supabase Realtime

GET /api/track/:token
  Description  : Public page — live location of user with this token
  Auth         : None (public, token-gated)
  Response 200 :
    {
      "user_name": "Priya",
      "last_lat": 12.9716,
      "last_lng": 77.5946,
      "last_seen": "2024-01-15T22:34:00Z",
      "audio_clip_url": "https://...",
      "sos_active": true
    }

PATCH /api/sos/:id/resolve
  Description  : Mark SOS event as resolved (safe arrival or admin resolve)
  Auth         : Required (user or admin)
  Request body : { "resolved_by": "user|admin", "notes": "string|null" }
  Response 200 : { "resolved": true }

--- THREAT & AI ---

POST /api/threat
  Description  : Report a threat signal from on-device sensors
  Auth         : Required
  Request body :
    {
      "user_id": "uuid",
      "module": "m1|m2",
      "signal_type": "audio|motion|eta|zone_exit",
      "confidence": 0.72,
      "lat": float,
      "lng": float

12. AI/ML COMPONENT SPECIFICATIONS
================================================================================

12.1 YAMNET — AUDIO THREAT DETECTION
--------------------------------------
  Model        : YAMNet (Google) — TFLite quantized, ~3.7MB
  Input        : 16kHz mono audio, 3-second window
  Output       : 521-class probability vector
  Target classes:
    - Screaming (class 20)     → threat if score >= 0.70 (M1) / 0.55 (M2)
    - Shouting (class 22)      → threat if score >= 0.65
    - Crying (class 21)        → threat if score >= 0.60
    - Glass breaking (class 60)→ threat if score >= 0.75
  Loop         : Sample every 10 seconds, infer on each sample
  On-device    : Yes — no audio uploaded until SOS fires
  Library      : TensorFlow Lite React Native bridge

12.2 THREAT FUSION ENGINE
--------------------------
  Language     : Python (FastAPI worker)
  Algorithm    : Weighted linear combination
  Inputs       :
    audio_score     (0–100) weight 0.40
    motion_score    (0–100) weight 0.25
    eta_score       (0–100) weight 0.20
    zone_exit_score (0–100) weight 0.15
  Time penalty : Add +15 if hour between 22:00–05:00
  Module delta : Subtract 15 from all thresholds for Module 2 (child)
  Output       : composite_score (int 0–100) + recommended_action string

12.3 ROUTE RISK SCORER
-----------------------
  Model        : GPT-4o-mini (OpenAI)
  System prompt:
    "You are a women's safety analyst for Indian cities. Given a destination
     coordinate, time of day, and nearby crime incident data, return a JSON
     risk assessment. Be specific and practical."
  Input data   :
    - Destination coordinates
    - Current time + day of week
    - NCRB crime CSV filtered to destination district
    - Google Places nearby (police stations, hospitals, metro)
  Output (JSON):
    { risk_score: 1–5, risk_label: string, reasoning: string,
      safe_alternative: string, nearby_safe_zones: [...] }
  Latency target: < 2 seconds

12.4 MINIML GROOMING CLASSIFIER
---------------------------------
  Model        : paraphrase-MiniLM-L3-v2 (quantized ONNX, ~17MB)
  Runs         : On-device (child's phone)
  Input        : Conversation metadata + flagged phrase clusters
                 (NOT raw message text)
  Features     :
    - Message frequency per hour (binned)
    - Contact age estimate (from name patterns)
    - Platform type (encrypted = higher weight)
    - Phrase category flags (secrecy, gift, isolation, migration)
    - Time-of-day distribution of messages
  Output       : grooming_score (float 0–1), flags (string[])
  Privacy      : Only score + flags uploaded to server, never raw text

12.5 DBSCAN HEATMAP CLUSTERING
--------------------------------
  Library      : scikit-learn DBSCAN
  Input        : All sos_events lat/lng from last 30 days
  Parameters   :
    eps         = 0.005 (roughly 500m at Indian latitudes)
    min_samples = 3
  Output       : Cluster centroids + radii → risk zones for authority map
  Runs         : Nightly cron job + on-demand via /api/admin/heatmap
  Storage      : Results cached in Supabase as materialized view

12.6 GPT-4O-MINI WEEKLY DIGEST
--------------------------------
  Prompt template:
    "You are a child safety advisor. Given the following weekly data for a
     child [age], write:
     1. A plain-English 3-sentence summary for the parent
     2. A natural conversation starter to discuss concerns without alarm
     Format as JSON: {narrative, conversation_starter}"
  Input        : composite_risk, top_flags, behaviour anomalies, content flags
  Output       : narrative (string), conversation_starter (string)
  Tone         : Supportive, not alarmist. Empowers parent, doesn't panic.

12.7 BEHAVIOUR BASELINE TRACKER
---------------------------------
  Language     : Python (scheduled worker)
  Algorithm    : Z-score anomaly detection (scipy.stats.zscore)
  Baseline     : 4-week rolling average per app per child
  Flag trigger : |z_score| > 2.0 on any metric
  Metrics tracked:
    - Daily screen time per app
    - Late-night session frequency (after 10pm)
    - New contact message volume
    - App switching frequency (anxiety indicator)

22. RISK & MITIGATION
================================================================================

RISK 1: YAMNet false positives (music, TV triggering alert)
  Mitigation : Require 2 consecutive positive samples before flagging
               Add motion correlation (sound + stillness = lower threat)
               User can dismiss false alarm within 10s confirm window

RISK 2: Background audio loop drains battery
  Mitigation : Sample every 10s (not continuous)
               Reduce to every 30s when journey mode is off
               Use expo-task-manager for efficient background execution

RISK 3: Twilio latency on first SMS
  Mitigation : Pre-warm Twilio by sending test ping at app launch
               Use Twilio Messaging Service (not direct from-number) for speed
               Target < 3s SMS delivery, < 8s total chain

RISK 4: Child resents monitoring (Module 3)
  Mitigation : App is transparent — child knows monitoring is on
               Parent dashboard shows summaries, not raw data
               Framed as safety, not surveillance — child sees their own score too

RISK 5: MiniLM grooming classifier false positives
  Mitigation : Require 3+ flag types before alerting parent (not just 1)
               Parent receives "conversation starter" not "your child is in danger"
               Score shown as trend over time, not binary alert

RISK 6: Supabase Realtime connection drops during demo
  Mitigation : Implement reconnection logic with exponential backoff
               Pre-seed demo data so dashboard looks populated even if live fails
               Have a 30s recorded backup video

RISK 7: Expo background task killed by OS (iOS)
  Mitigation : Use expo-task-manager with background fetch
               Request background location permission explicitly
               Fallback: foreground service notification keeps app alive

RISK 8: API cold start on Railway
  Mitigation : Keep minimum 1 instance always running (Railway setting)
               Pre-warm by pinging /api/health at app launch
               Health check endpoint returns 200 immediately
